# Deliberate — a product toolkit inside your coding agent

**Deliberate is a suite of essential, files-first product tools that run inside your coding agent** — for product managers, founders, business development, marketing, and any adjacent role. It can analyze any consequential idea or signal—from a feature to a change in marketing, strategy, or the platform and ecosystem—build testable prototypes where meaningful, monitor the competitive and market landscape, turn configured metrics and customer evidence into product readouts, and produce deep single-competitor teardowns. Each result is a grounded, reviewable Markdown document you can hand to a team, drop into a deck, or give to your own coding agent to act on. Everything is plain Markdown/JSON in the user's repo, all under `deliberate/`; there is no database and no cloud, and the records are versioned by the user's own git. **The project is simply the repo the harness is running in.**

**The toolkit at a glance:** `init` grounds a project once (the product context every tool reads); `case` analyzes an unresolved idea or signal and produces a scored decision record; `brief` tracks what changed in the competitive + market landscape; `readout` reports what product data and customers are saying; `matchup` tears down one named rival head-to-head into a battlecard; `source` curates durable grounding outside the current project folder; `address` works through a reader's in-record comments. Briefs, readouts, and matchups are discovery funnels into cases: each separates routine direct action from the consequential unresolved decisions worth analyzing. Each tool has its own section below.

**Deliberate vs. Sonorance.** *Sonorance* is the stable local app/platform Deliberate runs on: an agent-review workbench over Markdown for reading agent output, inspecting git diffs, leaving anchored comments, and editing when needed. *Deliberate* is the product toolkit that adds product-management capabilities per project without replacing Sonorance's engine or identity. Meet Deliberate primarily inside the coding agent through `/deliberate`, and optionally open the same files in Sonorance with `LAUNCHER serve --open`. Existing editor/file features are enabling mechanics; use mature libraries for commodity Markdown/file-management work and do not expand generic editor parity without telemetry-backed demand.

**What a `case` leaves on disk.** The deliverable is one combined decision record — `deliberate/cases/<YYYY-MM-DD-slug>/analysis.md` (frontmatter records the inferred `lens`; the case's **title** is the `# H1`; a 1–2 sentence **case summary** sits just under it; then a section per stage, in funnel order). The lens-aware go/no-go **score** is a recomputable companion — `score.md` beside the record, linked from a `## Score` section. Every case also writes `one-pager.md`: a customer-voice reverse PR-FAQ for product cases, a market decision brief for market cases, a strategy memo for strategy cases, or a platform decision memo for platform cases. A testable `prototype/[<surface>/]index.html` companion may be added to product and market cases on request; no prototype is forced for strategy or platform cases. **A case is just a prompt** — it is *not* saved as its own file; the workflow condenses it to the summary and keeps only that. The project context that grounds all of this lives in `deliberate/context/product.md` (+ `competitors.md` + `ecosystem.md`).

## Setup (do this first, every invocation)

The engine is driven through a launcher. Define it once for this session:

```
LAUNCHER = node .github/skills/deliberate/scripts/deliberate.mjs
```

Run everything below as `LAUNCHER <args>`. It needs Node ≥ 22. The stage reasoning runs in this session; the LLM-free engine builds prompts, validates configuration, persists artifacts, and extracts structured values. If a command prints "engine not found", tell the user to run `npx deliberate-cli install`, or set `DELIBERATE_ENGINE`.

## Commands

| Command | What you do |
|---|---|
| `help` | Show the current `/deliberate` command grammar, generated directly from the command registry. |
| `init` | Set up the current repo as a Deliberate project: **you** read its files directly, use durable internal or public resources outside the folder, and write the grounded context in `deliberate/context/` (`product.md` + `competitors.md` + `ecosystem.md`); the root README points agents to it. |
| `case <idea>` | Infer the decision lens, analyze the idea or signal through frame → shape → launch with the matching methods/templates, score the finished analysis with recorded evaluator provenance, and write the lens-appropriate one-pager. |
| `case <id> score` | (Re)compute the go/no-go after the full analysis; prefer an isolated cross-vendor Evaluator, record the actual model and independence status in `score.md`, and visibly label any same-session fallback. |
| `case <id> prototype` | Build (or rebuild) the testable prototype for an eligible completed product or market case; a recomputable companion, built on request. |
| `brief` | Produce a landscape **brief** — the competitive + market changes since the last brief (≤ 3 months). `brief list` lists them. |
| `readout [period]` | Produce a **product readout** grounded in one completed reporting period: the previous calendar week by default, or a natural-language override such as `for June` or `for Q2`. `readout list` lists them. |
| `matchup <competitor>` | Produce a competitive **matchup** — a grounded, point-in-time head-to-head against one named rival (one canonical doc per rival, refreshed in place). `matchup list` lists them. |
| `case list` | List cases with their lenses + scores. Change a case's state/title by editing its `analysis.md` (state/lens in frontmatter, title in the `# H1`); delete one by removing its folder — no sub-commands. |
| `source` | `source list` lists durable context sources outside the current project folder; `source add <location> ["<description>"] [--section <section>]` and `source remove <location>` manage internal or public resources recorded by category in `.sonorance/sources.md`. Project files are read directly and never added as sources. |
| `feedback` | Send the user's feedback to the makers — a bug (with repro), an idea (framed as the problem), or praise. You **structure, never rewrite**; their words go verbatim. See the `feedback` section. |
| `address` | Work through the reader's in-record comments in the open Sonorance app — answer or edit, then resolve (optional; only when the app is open). |

Routing: if the first word is one of these, follow its section. A bare `/deliberate <idea>` (no command) = `case <idea>`. If there's no project yet (no `deliberate/context/product.md`), do `init` first for every command except `help`. **A `case` runs the entire funnel in one pass** — there are no per-stage verbs to re-trigger; to change a finished record the user leaves in-record comments in the app and you `address` them (edit the file as a whole). After you produce or update any record, point the user at its file path (and, if the app is running, they read it there).

**Every substantive workflow ends with one clear next-step question, using the harness's native confirmation or multiple-choice UI and the generated follow-up contract sourced from `SKILL_FOLLOW_UPS`.** The CTA must name what will happen and why it is useful, not merely ask whether to "continue": `init` → run the first brief (default yes); `brief`, `readout`, and `matchup` with case-worthy signals → open the saved result for review in Sonorance while running the recommended cases (default), open it without running cases, or choose another action; product/market `case` → build the appropriate prototype (default yes); strategy/platform `case` → review the completed decision record in Sonorance (default yes); `prototype` → open it for review in Sonorance (default yes); `source add|remove` → refresh affected project context (default yes); `address` → review the resolved changes in Diff mode (default yes). Listing and feedback commands may end transactionally when no responsible next action exists.

## `init` — set up the project context (the Initiator)

A grounded `deliberate/context/` grounds every case, brief, product readout, and matchup. **You (the Initiator) derive it yourself** in this session (read the repo, fetch URLs) — the engine runs no model. Keep the exchange concise: group related choices at each confirmation gate and never interrogate the user one source at a time. The full method is a role you fetch with `init prompt`; this is the orchestration.

1. **Scaffold and expose the context.** Run `LAUNCHER init` → creates `deliberate/` with three markdown scaffolds (`context/product.md` + `context/competitors.md` + `context/ecosystem.md`) and makes this current repo the project. It idempotently creates or updates the root README (case-insensitive README variants are respected) with a **Product context for agents** section linking `deliberate/context/` and telling agent harnesses to read it before product, strategy, market, positioning, or implementation work. It also creates the shared platform config in `.sonorance/`; only `.sonorance/local/` and hidden `deliberate/` runtime subfolders are added to an existing `.gitignore`.
2. **Collect durable grounding outside this project folder — manual first, then propose more.** Read every relevant file inside the current project directly as automatic context. Never propose, confirm, or persist an in-project file or folder as a source, whether it is given as a relative path, absolute path, `file:` URL, or symlink. `.sonorance/sources.md` is for project-external resources, not necessarily organization-external ones: internal/private sources are expected and public sources are welcome.
   - **Ask.** *"Point me at durable material outside this project folder that describes the product or accumulates evidence over time. Internal resources are especially useful: product dashboards or queries, document folders, another repository, CRM/support/customer-feedback systems, research repositories, or issue trackers. Public sites, docs, changelogs, and APIs are welcome too. Paste any URLs or outside paths you have, or say 'none'."* For **each qualifying location**, run `LAUNCHER source add "<location>" ["<description>"] --section <section>`, choosing `product-strategy`, `code-delivery`, `metrics-data`, `customer-voice`, `go-to-market`, or `other`, then read it yourself in-harness. If the user names a file inside the current project, read it directly and do not run `source add`.
   - **Discover durable sources by section, not as one undifferentiated pile.** For each standard section (`product-strategy`, `code-delivery`, `metrics-data`, `customer-voice`, `go-to-market`), deliberately seek the few project-external resources that best ground that section. Prefer systems and corpora with recurring signal—folders/collections, repositories, live dashboards or queries, CRM/support/feedback systems, issue trackers, research repositories—and maintained canonical documents. Never propose an isolated feedback file, one support case or interview, a point-in-time metric export, or another short-lived small data point. Accept fewer—or none—when the project files and a smaller authoritative set are sufficient; never pad with weak, duplicate, stale, or incidental links. Inspect the repo (README links, manifest homepage/repository/docs fields, `docs/`, website/status badges) to discover and verify destinations, but never list the in-project files themselves as candidates. Verify every candidate before showing it; never fabricate a URL. Assign each accepted location to its strongest primary section and mention secondary relevance in its description.
   - **Public GitHub Issues are customer voice.** For every product-owned GitHub repository among the manual or discovered sources, verify whether it is publicly accessible and Issues are enabled. If so, propose its `/issues` page under `customer-voice`, read the issue signal where accessible, and include it in `product.md` → **Customer voice**. Never infer accessibility from a URL alone and never treat a private or disabled Issues page as evidence.
   - **Confirm before adding (default: accept).** Show **every auto-discovered project-external candidate**, grouped by section, as a bullet in this exact readable shape: `- <location> - <source description>`. Never include an in-project file or isolated point-in-time observation and do not collapse the list into counts or prose. Ask whether to *add them alongside the manually supplied sources* (**the default**), *keep only the ones they gave*, or *change the list*. Then run `LAUNCHER source add "<location>" "<description>" --section <section>` for each accepted source—**never drop or overwrite manually provided ones**—and read each in-harness. Manual-only → add nothing further.
3. **Get the method, then fill the files.** Run `LAUNCHER init prompt` → a `MODEL:` line, a `===== SYSTEM =====` block (AGENTS.md + the **Initiator instructions**) and a `===== TASK =====` block (the attached project-external sources + the three current scaffolds). **Follow it:** read the current project directly + each attached source, then **edit `deliberate/context/product.md`, `competitors.md`, and `ecosystem.md` directly**, replacing every section's guidance with real, grounded markdown. Guardrails the method insists on:
   - **Competitors are research, not fabrication: exhaustive discovery, then strict qualification.** Search direct category rivals, cross-category commercial products/services for the same buyer/job/budget, overlapping suites/platforms, and emerging/niche/regional/open-source products through at least two independent discovery routes per lane. Continue full passes until another pass produces no new qualified name. Include every active named competitor that has current overlap evidence, a material reason it matters, and at least one verified official source; use no numeric target or cap. Exclude manual workflows, spreadsheet workarounds, internal processes, "do nothing," and other status-quo substitutes. Put the complete qualified roster only in `competitors.md`; `product.md` keeps only its link.
   - **Ecosystem is broad discovery, then strategic qualification.** Put the complete roster, details, and monitoring sources only in `ecosystem.md`; `product.md` keeps only its link. Search each position (**adjacent / complement / channel / mover**) through at least two independent routes until another full pass yields no new qualified player; include every player with a specific relationship or analogy, material decision relevance, current first-party evidence, and a useful official monitoring source. A **complement** is a distinct product or service used before, alongside, or after this product in the same end-to-end workflow where co-use or interoperability raises the value of both for shared users; it is not a substitute for the primary job, a dependency included for that reason alone, or any generic tool the same user uses. Intentionally search other markets and niches for adjacent analogs with similar strategy, packaging, GTM, goals, technology, or product dynamics.
   - **Missing info → never guess.** If the sources don't cover a section, ask the user to add a source or provide the details; if still unavailable, mark it `Not covered by the provided sources — add a source or fill in manually.` — never "unknown", never fabricated.
4. **Confirm the risky bits.** Show the user the **personas**, the complete qualified **competitor** and **ecosystem** rosters, genuinely uncertain borderline candidates separately, and the **readout cadence/alignment/timezone**, then ask them to confirm/correct just those; update to match. Never hide a long qualified roster behind a count or truncate it for brevity.
5. **Recommend the hero next step.** Report that context is saved to `deliberate/context/` and linked from the root README, briefly explain that the first brief establishes a current market/competitive baseline and surfaces grounded opportunities worth analyzing, then ask with the harness's native confirmation: *"Run the first landscape brief now?"* with **Run brief** (default) and **Not now**. On approval, run the `brief` workflow below; do not steer directly to a case after init.

## `case <idea>` — analyze an idea or signal and run the funnel

**You (the host agent) run the pipeline in THIS session** with three roles:

- **Analyst — YOU, in this session** (not a sub-agent). One context-rich author writes `frame` (decision, evidence, current state, constraints, stakes), `shape` (alternatives, tradeoffs, recommendation, boundaries), and `launch` (concise next commitments, proof, sequence, and a revisit condition) through the recorded lens. **Keep this reasoning in your own context** so refining a result reuses *why* you wrote it, instead of a fresh agent rediscovering everything.
- **Evaluator — preferably an isolated sub-agent** on the score's cross-vendor `MODEL:` target. It judges the complete analysis and flags ungrounded claims. If the harness cannot isolate it, an inline fallback is allowed only with honest provenance: the save omits `--independent`, and `score.md` visibly says it is not an independent second opinion.
- **Prototyper — YOU** (or a sub-agent), building a product journey prototype or market test artifact on request once the analysis is complete. No prototype is available for strategy or platform cases.

`<idea>` may be free text, a URL, or a file path. If a URL or path outside the project describes durable *product/market* context rather than one isolated observation, add it as a source first (`LAUNCHER source add "<location>"`); if it is inside the current project, read it directly without adding it. Otherwise fold its gist into the idea text.

1. **Infer and create the case.** Determine the primary commitment the decision would make: **product & experience** (`product`) for a customer capability or experience; **market & commercial** (`market`) for audience, positioning, pricing, packaging, distribution, launch, or sales; **strategy & portfolio** (`strategy`) for where to play, how to win, priorities, portfolio, resources, or capabilities; **platform & ecosystem** (`platform`) for build/buy/adopt/integrate/migrate/partner/standard/dependency choices. A case may touch several domains, but choose the lens of its primary decision and let the other methods support it. If genuinely ambiguous, ask one focused question about the decision to unlock; never ask the user to operate a lens switch. Then run `LAUNCHER case "<one-line idea>" --lens <product|market|strategy|platform>` as an internal host action and retain its printed id as `<case-id>` for every later command. The user-facing command remains `/deliberate case <idea>` with no required switch. The raw idea is held only as a transient input; the recorded lens is durable.
2. **Give it a concise title** (there is no title agent — you do it): edit the record's `# H1` in `deliberate/cases/<YYYY-MM-DD-slug>/analysis.md` to a short, specific title — **≤ ~10 words, < 50 chars**. The title is that `# H1` (there is no title field and no retitle command — you just edit the file). The folder slug keeps its provisional value; that's fine (folders sort by date, and the title lives in the body).
3. **Summarize the case** (1–2 short sentences): condense the original idea into a plain-English summary a reader sees directly under the title — what the case is about, grounded in what the user actually wrote. **Write it directly into `analysis.md` as the lede** — the prose line(s) between the `# H1` and the first `##` section (no command; it's a plain file edit, kept on one logical line, not hard-wrapped). Once that lede exists the engine **drops the raw prompt** on the next save (a case is "just a prompt"): from here on the record keeps only the summary, and later stages ground on the summary + the frame. *(Do this right after producing `frame` below, so the frame still grounds on the verbatim prompt — or immediately if you've already captured the idea faithfully.)*
4. **Run the lens-aware analyst funnel** — repeat until the analysis is **complete**, walking `frame → shape → launch`; the engine reads the recorded lens and selects the matching method, supporting skills, and template at every stage:
   1. **Get the prompt:** `LAUNCHER case analysis prompt <case-id>` → a `MODEL:` line, a `===== SYSTEM =====` block (AGENTS.md + project context + the stage's skills + instructions) and a `===== TASK =====` block (the accumulated per-case context + the output template).
   2. **Produce the artifact yourself, in this session** (the Analyst) — the reasoning must persist for later refinement (don't spawn a throwaway sub-agent). Ground strictly in the SYSTEM/TASK content; never invent demand or evidence.
   3. **Persist:** write the artifact to a file → `LAUNCHER case analysis save <case-id> --file <path>` (or pipe via stdin). It appends the stage's section to the single combined `analysis.md` (headings are normalized into one clean hierarchy — don't add your own summaries or attribution) and advances the case; after `launch` it prints **`analysis complete`** — the funnel is done.
5. **Score the complete analysis with honest provenance.** The go/no-go is a recomputable `score.md` companion, not a funnel stage. The Evaluator applies the case's lens-specific criteria; scores are advisory within that lens and must not imply that scores across different lenses are directly rankable:
   1. **Get the prompt:** `LAUNCHER case score prompt <case-id>` → the target `MODEL:` line and SYSTEM/TASK blocks with the finished record.
   2. **Prefer an ISOLATED sub-agent** on that cross-vendor model. Record the model that actually produced the artifact, not merely the requested target. If no isolation tool exists, do a fresh inline pass and treat it explicitly as a same-session fallback.
   3. **Persist an isolated result:** `LAUNCHER case score save <case-id> --model <actual-model-id> --independent --file <path>`.
   4. **Persist an inline fallback:** `LAUNCHER case score save <case-id> --model <actual-model-id> --file <path>` (no `--independent`). The generated `score.md` visibly labels the fallback as non-independent. Both forms stamp the number and link the companion from `analysis.md`.
6. **The analysis is done — write the Key highlights.** In ≤ 3 **short, concrete** bullets (an executive summary of the decision — no fluff), capture what a reader must know: the verdict + why, the single biggest risk or gap, and the strongest lever. **Add them directly to `analysis.md`** as a `## Key highlights` section placed at the top — right after the summary lede, before the first stage section (no command; a plain file edit). Keep **each bullet on one logical line — never hard-wrap prose across several source lines** (the same rule as the summary lede). The engine recognizes that heading and keeps it above the stages.
7. **Write the lens-appropriate one-pager** (always, as part of a case): Product → customer-voice reverse PR-FAQ; Market → market decision brief; Strategy → executive strategy memo; Platform → platform decision memo.
   1. **Get the prompt:** `LAUNCHER case one-pager prompt <case-id>` → a `MODEL:` line, a `===== SYSTEM =====` block (AGENTS.md + project context + the selected lens skills + One-pager instructions) and a `===== TASK =====` block (the finished record + lens-specific template).
   2. **Produce it yourself** (the Analyst — keep it in your own context; don't spawn a throwaway). Distil **only** the record — invent no capability, customer, price, metric, partner intent, resource, or technical fact; preserve the recommendation's honest boundaries and uncertainty; keep it to ~one page.
   3. **Persist:** `LAUNCHER case one-pager save <case-id> --file <path>` (or pipe via stdin). It writes `deliberate/cases/<YYYY-MM-DD-slug>/one-pager.md` beside `analysis.md` and links it from the record (a `## One-pager` section). Don't hand-edit that link — the engine regenerates it.
8. **When the analysis is complete, report the decision and ASK.** From the artifacts you just produced (the `score`, `shape`, and `launch` sections now in the record), present a compact **decision card**, grounded only in those artifacts:
   - the **score** and verdict (advance / shelve / reject),
   - the one or two **reasons** the score landed there, and the **evidence** each rests on,
   - the case lens and where the record is on disk (`deliberate/cases/<YYYY-MM-DD-slug>/analysis.md`, with the recomputable `score.md` and lens-appropriate `one-pager.md` beside it).

   **Point the user at the record** on disk; if they want to read it in the UI and the app isn't running, start it directly on the record with `LAUNCHER serve --open --file "<the printed deliberate/cases/.../analysis.md path>"`.

   For **Product or market cases**, ask for approval using the harness's native confirmation / multiple-choice prompt (not prose) — *"Build the product journey prototype now?"* for Product or *"Build the market test prototype now?"* for Market, with **Yes** (default), **No**, and **Other**. **Wait for the choice.** Don't print raw shell commands; don't proceed on your own.
   - **Yes:** build the prototype for this case now — the `prototype` flow below.
   - **No:** stop; the analysis is complete and the case waits — the user can `prototype` it later.
   - **Other:** read the reply, refine (below), then re-ask.

   For **Strategy or platform cases**, do not offer or attempt a prototype. Ask *"Open the completed decision record in Sonorance for review?"* with **Open for review** (default) and **Not now**; on approval run `LAUNCHER serve --open --file "<the printed deliberate/cases/.../analysis.md path>"`. These confirmations are host behavior, not engine gates: the analysis is already complete.

## Refining a finished record — comment, don't re-run

There are **no per-stage verbs**. A `case` runs frame → shape → launch in one pass and is scored separately with recorded evaluator provenance. The record is then edited as a whole, not by re-triggering stages. When the user wants changes, they leave in-record comments in the app and you `address` them. When an edit changes the analysis, re-run `LAUNCHER case score prompt <case-id>`, produce the evaluation, and save it with the same explicit id plus the actual model/provenance flags.

## `prototype <id>` — build a product or market test artifact

Produced **in-harness** (the Prototyper) for **product and market cases only**, right after analysis or any time later. Never auto-built; it is a companion built only on request. A product case gets one prototype per primary surface, with medium native to that surface. A market case gets a testable market artifact at the selected customer touchpoint—such as a landing/pricing page, message or campaign sequence, sales narrative, or partner offer. No prototype is available for strategy or platform cases; if selected, explain that their decision record and one-pager are the completed artifacts.

1. **Pick the case.**
   - If the user named one (`/deliberate case <id> prototype`), use it if its lens is product or market; otherwise explain that it has no prototype.
   - If neither is clear, **ask** with the harness's native multiple-choice, each option labelled `id` · title · score:
     - **Default** — the eligible product or market case you were just discussing; else the latest eligible completed case without a prototype (`LAUNCHER case list --state done`, newest first).
     - **Second** — the latest eligible case that scored `advance`, if different from the default.
     - **Third** — the latest eligible case overall, even if it already has a prototype, if different from the above.
     - **Other** — the user names a case.
2. **Build in-harness.** For a product case, read `product.md` → **Interfaces** and the connected implementation sources, then use `LAUNCHER case prototype prompt <case-id>` to build every shaped journey step in order on the default primary surface, in its native medium, with real product conventions and a reachable failure or recovery path; save with `LAUNCHER case prototype save <case-id> --file <path>` and use `--surface <slug>` for each additional primary surface. For a market case, use the same explicit-id prompt/save pair to build one focused customer-touchpoint artifact that exposes the grounded audience, claim, proof, action, observable response, and meaningful objection or alternate path; use `--surface <slug>` only to distinguish multiple grounded touchpoints. In both cases, return valid self-contained HTML that works from `file://`, uses realistic disclosed mock data, makes every promised interaction reachable, and performs no real network calls.
3. **Report** the built prototype(s) (`LAUNCHER case prototype list <case-id>`) — `deliberate/cases/<YYYY-MM-DD-slug>/prototype/[<surface>/]index.html` (the combined `analysis.md` links each one under `## Prototype`). The prototype is a recomputable companion — rebuild any surface any time to refine it.
4. **Offer review.** Ask with the harness's native confirmation: *"Open the prototype in Sonorance to review it against the case?"* with **Open for review** (default) and **Not now**. On approval, run `LAUNCHER serve --open --file "<the case's deliberate/cases/.../analysis.md path>"`; the record links the prototype and gives the reviewer the decision context.

## `brief` — the periodic landscape brief (the Briefer)

A **Brief** is a project-scoped landscape report (not a case): the **competitive + market changes since the last brief**, capped at 3 months. You (the host) are the **Briefer** — you do the research yourself in this session (fetch changelogs, blogs, release notes, GitHub activity, roadmaps; scan the market), then distil it to only what a founder / PM / manager would act on, every finding grounded in a linked source. There is no evaluator and no gate — a brief just surfaces signal.

1. **Get the prompt:** `LAUNCHER brief prompt`. It prints a `MODEL:` line (produce in THIS session), a `===== SYSTEM =====` block (AGENTS.md + project context + `competitors.md` + `ecosystem.md` monitoring sources + the `landscape-scan` skill + Briefer instructions) and a `===== TASK =====` block with the **reporting window** (`period_start → period_end`), the **previous brief** (read-only prior context, when one exists — so you report only what changed *after* it and never re-report it), and the output template. **The window is the hard boundary** — *since the last brief*, at most 3 months back (a first-ever or stale brief → a full 3-month window; a later brief is told so explicitly and carries the prior brief).
2. **Research the window, yourself.** For **each named competitor** in `competitors.md`, scan their **first-party** signals *inside the window* — announcements, product blog, release notes / changelog, roadmap changes, public codebase activity (notable commits / merged PRs), pricing moves. Capture **≤ 3** meaningful highlights each, **every one with a working source link**; a competitor that was quiet gets exactly **"No meaningful updates."**. Then scan the wider **market & ecosystem** — the named ecosystem players from `ecosystem.md` (adjacent / complement / channel / mover moves), the Market section in `product.md`, protocol / standard releases, new entrants or adjacent players, M&A / funding / partnerships / pivots, new technologies — and keep the **≤ 3** most important, actionable developments, each with a source.
3. **Distil the decision.** Pick the **top three (fewer is better)** findings across both lenses for **Key highlights**. Each must name the actor/product, the concrete change and date/evidence, and why it matters specifically to this product; never replace specifics with a vague category trend. Then reason the **Action items** and label each follow-through as product case, market case, strategy case, platform case, or Direct action. Only consequential unresolved choices become cases; routine execution, instrumentation, already-decided work, and bounded research stay direct. Keep every bullet **≤ 3 sentences (ideally 1–2)**.
4. **Persist:** write the brief to a file → `LAUNCHER brief save --file <path>` (or pipe via stdin). It records the window and writes `deliberate/briefs/<YYYY-MM-DD>/brief.md`, printing its id.
5. **Report a decision-ready brief, not a headline list.** State the reporting period and file path. For each Key highlight, summarize the named actor/product, what concretely changed, the dated/source-backed evidence, and why it matters to this product. Then summarize each warranted Action item and the decision it would unlock. For every recommended case candidate, name its inferred lens, concise problem/decision title, triggering evidence, why analyzing it is valuable now, and the decision it would help make. Do not make the user open the file just to understand the recommendation. If the user asks to review the brief in Sonorance, run `LAUNCHER serve --open --file "deliberate/briefs/<the saved id>/brief.md"`.
6. **Open the result and run the relevant cases by default.** When case candidates exist, ask with the harness's native multiple-choice prompt: *"How should we follow up on this brief?"* with **Open for review + run cases** (default), **Open for review only**, and **Other**. For the default, start `LAUNCHER serve --open --file "deliberate/briefs/<the saved id>/brief.md"` as a background process so review is immediately available, then continue without waiting and run one full case per recommended candidate, phrased around the underlying problem or decision rather than a predetermined solution. **Open for review only** starts Sonorance on the saved brief and does not run cases. **Other** lets the user specify another action, including selecting only some cases. Offer a prototype only for product or market cases. Do not propose a case for direct actions. When no case candidate exists, offer to open the brief for review without manufacturing one.

Ground everything: no source link, no highlight; never fabricate funding, partnerships, competitors, or releases; never re-report a change a prior brief already covered.

## `readout [period]` — the periodic product readout (the Reporter)

A **product readout** is a project-scoped evidence synthesis (not a case and not a landscape brief): what the configured metrics and customers are saying, what materially changed, and what deserves action. You (the host) are the **Reporter** — inspect and query the project's configured evidence yourself in this session, then produce a concise executive read with enough detail to audit every conclusion. One completed reporting period grounds the entire report. Read `product.md` → **Metrics & traction** for the durable readout cadence, calendar alignment, and timezone; init defaults it to the previous completed Monday–Sunday calendar week. If the user adds a natural-language override such as `/deliberate readout for June`, `/deliberate readout for Q2`, or an explicit completed date range, use that completed calendar period for this run. If the requested period is ambiguous or still in progress, ask for clarification or explain that only completed periods can be analyzed.

1. **Resolve one completed period.** Convert the configured default or user override to inclusive ISO dates in the project's timezone. Run `LAUNCHER readout prompt --period-start <YYYY-MM-DD> --period-end <YYYY-MM-DD> --timezone <IANA timezone>`. The engine rejects a range whose end date is today or later in that timezone. Use the exact same dates and timezone on `readout save`. The prompt includes the immediately preceding equivalent completed comparison period; a calendar month compares with the prior calendar month, a quarter with the prior quarter, and an arbitrary range with the immediately preceding equal-duration range.
2. **Check evidence coverage before interpreting it.** Read every relevant configured source you can access, prioritizing **Metrics & data** and **Customer voice**, then product/strategy, delivery, and go-to-market context. A source can be a URL, local file, dashboard, query, warehouse, support export, CRM, transcript, or another harness-accessible location. Never imply a source was inspected when it was inaccessible, stale, empty, or outside the requested permissions; carry that limitation to **Data gaps**.
3. **Use the reporting period for everything.** Metrics, customer evidence, releases, experiments, incidents, Key takeaways, Insights, and Actions must use only evidence inside the report-level period. Calculate every metric over that period using its durable definition, source/query, aggregation, and segment; compare it with the supplied immediately preceding equivalent period. The period appears once at the top of the report, so the metric table needs only **Metric · Value · Comparison · Read**. Never mix in the period currently in progress, project or annualize partial data, use the previous readout artifact as the comparator, or add a target.
4. **Render material trends when history supports them.** Add no more than three charts, only for decision-relevant metrics with at least four comparable completed periods (prefer six to twelve) at the readout's configured cadence. The table remains canonical; charts reveal the longer trajectory. Never smooth, interpolate, forecast, add target lines, mix period grains, or chart sensitive detail that should not be committed. Create a temporary bundle containing `readout.md` and `charts/`. For each chart, write a JSON spec shaped as `{"title":"Weekly active teams","unit":"teams","comparison":"WoW","source":"<traceable source>","series":[{"period_start":"YYYY-MM-DD","period_end":"YYYY-MM-DD","value":123}]}` with four to sixty strictly ordered points, then run `LAUNCHER readout chart --spec <json> --output <bundle>/charts/<safe-metric-name>.svg`. Embed it directly below Key metrics as `![descriptive metric trend](charts/<safe-metric-name>.svg)`. If no series qualifies, generate no chart and keep the report Markdown-only.
5. **Synthesize the read.** Lead with at most three Key takeaways. Keep Key metrics compact but complete; put cross-metric and customer interpretation under Insights; use exact representative customer quotes only when they are verifiable, relevant, dated/source-linked, prevalence-backed, and privacy-safe. Distinguish observation, context, hypothesis, and causal evidence. Turn only decision-relevant findings into Actions with an owner/decision, timing, and success signal; do not force positive/negative symmetry, invent a health score, dump raw metrics, or prescribe work unsupported by the evidence.
6. **Persist:** when charts exist, run `LAUNCHER readout save --bundle <bundle> --period-start <YYYY-MM-DD> --period-end <YYYY-MM-DD> --timezone <IANA timezone>` so `readout.md` and every referenced SVG are stored atomically. Without charts, use `LAUNCHER readout save --file <path> --period-start <YYYY-MM-DD> --period-end <YYYY-MM-DD> --timezone <IANA timezone>` (or pipe via stdin). Use the exact same dates and timezone as `prompt`; save rejects a mismatched `Period:` line. Each run writes a new `deliberate/readouts/<YYYY-MM-DD[-N]>/readout.md`; same-day runs remain separate artifacts.
7. **Report** the Key takeaways, Actions, material Data gaps, and the file path. `LAUNCHER readout list` lists the project's readouts. If the user wants the UI and it is not running, start `LAUNCHER serve --open --file "deliberate/readouts/<the saved id>/readout.md"` so the completed readout opens immediately.
8. **Open the result and offer case-worthy follow-through.** For each Action that represents a consequential unresolved decision, infer and name the product, market, strategy, or platform case lens, then give the signal, why analysis is valuable, and the decision it would unlock. When candidates exist, ask *"How should we follow up on this readout?"* with **Open for review + run cases** (default), **Open for review only**, and **Other**. For the default, start `LAUNCHER serve --open --file "deliberate/readouts/<the saved id>/readout.md"` as a background process, then continue without waiting and run one full case per recommended candidate. **Open for review only** starts Sonorance on the saved readout and does not run cases. **Other** lets the user specify another action, including selecting only some cases. Routine operations, instrumentation fixes, already-decided work, and bounded research remain direct actions. When no case candidate exists, offer to open the readout for review without manufacturing one.

Reliability is explicit: incomplete evidence yields a useful bounded readout plus visible gaps, never a confident fiction. If there is no accessible metric or customer evidence, do not pad the template; explain that a grounded readout cannot yet be produced, identify the minimum sources/definitions needed, and do not save a success-shaped artifact. Chart rendering is deterministic and optional; a chart is never evidence by itself and never compensates for missing or inconsistent source data.

## `matchup` — the single-competitor head-to-head (the Scout)

A **Matchup** is a project-scoped, single-rival **head-to-head** (not a case, not a brief): a full, honest, sourced read of **one named competitor** against us across every dimension — strengths, weaknesses, gaps, journeys, JTBD coverage, positioning, reuse/interop, and the "why we win / why we lose" a competing deck is built from. You (the host) are the **Scout** — you research the rival yourself in this session, then write the definitive current read, grounded in linked sources and true **as of** the run date. There is no evaluator and no gate. Where `brief` is *breadth of change over time*, `matchup` is *depth on one rival at a point in time*.

**Keyed by the rival, refreshed in place:** there is exactly one canonical `matchup.md` per competitor (`deliberate/matchups/<slug>/matchup.md`). Re-running the same rival **updates that doc**, keeping its id and creation date and restamping the as-of date (git carries the history) — it never spawns a dated copy.

1. **Get the prompt:** `LAUNCHER matchup prompt "<competitor>"` (a name, product, or URL). It prints a `MODEL:` line (produce in THIS session), a `===== SYSTEM =====` block (AGENTS.md + project context + `competitors.md` + the `head-to-head` / `jtbd` / `positioning` / `prioritization` / `tech-constraints` / `landscape-scan` skills + Scout instructions) and a `===== TASK =====` block naming **the rival**, **the as-of date**, the **existing matchup** (read-only prior read, when one exists — so you refresh it in place), and the output template.
2. **Research the rival, yourself.** Fetch their site, docs, changelog, repo, pricing, positioning. **Steelman them first** — establish honestly what they do better and why a smart buyer picks them — *before* finding the openings; a home-team read loses the deck in Q&A. **If the rival isn't already in the canonical roster in `competitors.md`,** research it first-party from scratch, then **ask the user whether to add it to the project context — default yes.** On yes, add its details and official monitoring sources only to `competitors.md` (the same edit `init` makes), so `brief` starts tracking it; on no, proceed without touching context.
3. **Read across every dimension** — user journey, UX, functional, supported scenarios, strategic, marketing & positioning, pricing & business model, distribution & GTM, implementation, ecosystem & interop, maturity & momentum — ending each with an honest **Edge: us / them / even — because…** ("even" is valid; don't manufacture contrast). Then **synthesize**: SWOT the pair (name our real weaknesses), a JTBD coverage grid, and the strategy-canvas value curve.
4. **Turn it into a decision** — a **battlecard** (why we win / why we lose / landmines / objection handling, ≤3 each with proof), **positioning against them** (the frame + the do-not-compete), and prioritized **opportunities** (Borrow by value ÷ effort — learn, don't copy, respect their license — Partner/interop, Respond). From the strongest actionable insights, identify at most three consequential unresolved decisions as case candidates, each with its inferred lens, triggering insight, value of analysis now, and decision unlocked. Routine execution and bounded research remain direct opportunities. Keep every bullet **≤ 3 sentences (ideally 1–2)**.
5. **Persist:** write the matchup to a file → `LAUNCHER matchup save "<competitor>" --file <path>` (or pipe via stdin). It cleans the artifact, stamps the as-of date, and upserts `deliberate/matchups/<slug>/matchup.md`, printing its id.
6. **Report** where it landed, the Bottom line, and each case candidate in decision-ready form: lens, underlying problem or decision, triggering insight, why analysis is valuable now, and what decision it would unlock. `LAUNCHER matchup list` lists the project's matchups (one per rival). If the user wants to read this one in the UI, run `LAUNCHER serve --open --file "deliberate/matchups/<the saved id>/matchup.md"` so it opens immediately.
7. **Open the result and offer case-worthy follow-through.** When the matchup contains one or more case candidates, ask *"How should we follow up on this matchup?"* with **Open for review + run cases** (default), **Open for review only**, and **Other**. For the default, start `LAUNCHER serve --open --file "deliberate/matchups/<the saved id>/matchup.md"` as a background process, then continue without waiting and run one full case per recommended candidate, phrased around the unresolved decision rather than a predetermined response. **Open for review only** starts Sonorance on the saved matchup and does not run cases. **Other** lets the user specify another action, including selecting only some cases. If there are no case-worthy decisions, offer to open the matchup for review without manufacturing one; direct opportunities remain direct work.

Ground everything: every factual claim links to a first-party source dated on or before the as-of date; mark confidence (fact ▪ inferred ▪ assumption); steelman the rival and name our real weaknesses; never fabricate features, pricing, funding, or adoption; and never bolt on a slide outline — the substance already seeds a competing deck.

## `case list` · `source` · reading in the app

- **`case list`** — `LAUNCHER case list` lists cases + ids + lenses + scores. There is no singleton active case and there are **no manual shelve/dismiss states** — a case is simply kept or deleted: to **rename** one edit its `analysis.md` `# H1`, to correct its inferred lens edit the `lens` frontmatter before generating more artifacts, and to **remove** one delete its folder.
- **`source`** — `LAUNCHER source list` lists durable project-external sources by section; `source add <location> ["<description>"] [--section <section>]` adds one and `source remove <location>` drops it. Standard sections are `product-strategy`, `code-delivery`, `metrics-data`, `customer-voice`, `go-to-market`, and `other`; users may also create custom `##` headings in `.sonorance/sources.md`. A `<location>` is an internal/private or public URL or local path outside the current project folder; in-project files are automatic context, rejected by `source add`, and omitted from the source list if a legacy entry remains. Curate durable systems, corpora, and canonical documents—not one-off feedback, support cases, interviews, exports, or other small point-in-time observations. Nothing is cloned or fetched. After adding or removing a source, explain which context sections it can affect and ask *"Refresh the affected project context now?"* with **Refresh context** (default) and **Not now**; on approval, read the source in-harness and update `product.md`, `competitors.md`, or `ecosystem.md` as needed.
- **Reading records in the UI** — we operate on **files**, not ids: point the user at the file path. If they want the web UI, `LAUNCHER serve --open --file "<project-relative Markdown path>"` starts Sonorance over the repo and opens that exact artifact. There is no open-by-id command.

The **project context** is three Markdown files you author and the user reads/edits directly: `product.md` for core product context and links to the other two; `competitors.md` as the only competitor roster/detail source; and `ecosystem.md` as the only ecosystem roster/detail source. There is no `context` command; just open the files.

## `address` — work through the reader's in-record comments (when the app is open)

The optional Sonorance app (`deliberate serve`) lets the user **select any span of any file in the project — a case's decision record, a landscape brief, a product readout, a matchup, or any other file — and leave a comment** (a question or a change request), GitHub-PR-review style. When the user has finished annotating they run `/deliberate address` (or ask you to); you then **work through the open comments** and resolve each. The port is auto-resolved from the app's `serve.json`.

1. **Fetch.** Run `LAUNCHER comment list` — it returns `{ comments:[ { id, file, anchor:{quote,heading}, body } ], count }`, the batch of every **open** comment across the project. Each comment names the **`file`** it annotates (a project-root-relative path). If the command exits nonzero, stop and surface its app/transport error and printed local diagnostics path; never treat that as an empty review queue. If `count` is 0, there are no open comments.
2. **Address each comment.** Open the comment's `file`, find the quoted span (`anchor.quote`) in its section (`anchor.heading`), then:
   - **A question** → answer it **here, in this conversation** (grounded in the file/context — the user reads your answer in this agent interface). No file change.
   - **A change request** ("drop this competitor", "tighten the urgency claim") → **before editing, confirm with the user** using the harness's native confirmation / multiple-choice prompt when the change is material or ambiguous (*"Apply this edit to `<file>`?"*). On yes, **edit that `file`** to make the change (a case's `analysis.md` round-trips through the app's diff view, reviewable before the user commits — the propose-and-commit ethos: you propose, their git commits).
   - **Keep the case's documents consistent.** A case has `analysis.md` and a lens-appropriate `one-pager.md`. When a change to one invalidates the other, revise both in the same pass. After a material analysis edit, regenerate the one-pager (`LAUNCHER case one-pager prompt <case-id>` → produce → `LAUNCHER case one-pager save <case-id>`) and refresh Score with `LAUNCHER case score prompt <case-id>` followed by a provenance-aware save carrying the same id. A cosmetic one-pager wording fix needs no analysis change.
3. **Resolve.** `LAUNCHER comment <commentId> resolve` (add `--revised` if you edited a file; add `--note "<what you did>"` to record a short note in the durable `comments.jsonl`). The comment disappears from the app the moment it's resolved.
4. **Loop** through the remaining comments, then tell the user what you answered / changed. If you edited a record, point the user at its file path.
5. **Offer review.** When files changed, ask *"Review the resolved changes in Sonorance Diff mode?"* with **Review changes** (default) and **Not now**. The app is already open for addressing; point the user to the revised files rather than starting another server.

Addressing is **optional and non-blocking** — only do it when the app is plausibly open. Never invent answers: an in-record answer is held to the same grounding bar as any stage artifact.

## `feedback` — send the user's feedback to the makers (structure, never rewrite)

When the user wants to report a bug, request something, or share praise, run `LAUNCHER feedback "<their message>" [--category bug|idea|praise|question|other] [--rating <1-5|up|down>]` — or pass a structured record with `--file <json>` (or pipe the JSON on stdin) when you've collected more than one field. The message is sent **exactly as the user framed it**; your job is to help them structure it, not to reword it.

**Your role is to STRUCTURE, never to REWRITE.** Preserve the user's own words verbatim (the founder distils the JTBD later, on their side — not you, not now). Do two light things before sending:
- **For an idea / feature request — steer them to the PROBLEM, not a solution.** Nudge the user to say *what they're trying to achieve and why* rather than prescribing the feature. If their message is only a proposed solution, ask one short question ("What are you trying to do, and what's getting in the way?") and put their answer in `--context`. If it still reads as solution-only, pass `--needs-framing` so the record is flagged — but **send their words unchanged**.
- **For a bug — get repro completeness.** Encourage steps-to-reproduce, expected vs. actual, and how often it happens; put these in the `bug` fields of a `--file` record. A pasted log or stack can go in `bug.diagnostics` — it is **scrubbed** (paths/secrets/emails stripped) before it leaves the machine.

**Privacy is absolute — this channel carries only what the user typed.** Never send prompts, model completions, file contents, or full logs. The one field that carries free text is the user's own `message` (and their `--context` answer), because they explicitly authored it to send. Optionally you may attach a short, safe **agent-drafted context summary** via `--agent-context` — a one-or-two-sentence recap of what the user was trying to do and the gap they hit — but only after you've **shown it to the user to approve, edit, or skip**, and it must never quote file contents, secrets, or anything session-specific that would reveal more than the user intends. Default: offer it for bugs (previewed), ask first for ideas.

After sending, confirm to the user that their feedback went through (the CLI prints a short id). The CLI sends **one** consented feedback record (never a duplicate telemetry event), correlated with product usage by the same anonymous install id, and writes a durable local mirror in the project's `.sonorance/local/`.

**A note on telemetry (be transparent when asked).** Deliberate and Sonorance emit content-free OpenTelemetry product usage and error signals to help improve the product — never prompts, completions, file contents, paths, or file/folder names. It is controlled from Sonorance Settings, the `sonorance` binary, `SONORANCE_TELEMETRY`, and a committed `.sonorance/config.json` `"telemetry": false` team policy. Telemetry is not a `/deliberate` command.

## Rules

- **The prototype is eligible, asked for, and never automatic.** Offer it only for product and market cases, after analysis, and wait for approval; build it in-harness with `case prototype prompt <case-id>` → `case prototype save <case-id>`. Cases under the strategy and platform lenses end in their decision record + one-pager and must not be forced into a prototype. This confirmation is host behavior, not an engine state.
- **Evaluator provenance is honest.** Prefer an isolated cross-vendor evaluator after the complete analysis. Save with `--independent` only when isolation actually occurred; otherwise omit it so `score.md` visibly labels the same-session fallback. Always pass the actual evaluator model with `--model`.
- **Ground everything.** Every claim traces to the three files in `deliberate/context/`, the case's artifacts, project-external sources, or the repo. Never invent demand, personas, or competitors.
- **Files are the deliverable.** The records live in the repo (`deliberate/context/product.md`, `deliberate/…`), versioned by the user's own git. Point the user at those files; the Sonorance app is an optional reader, never required.
- **Be concise.** A run header, a streamed sense of progress, and a tight decision card — not a wall of text.
